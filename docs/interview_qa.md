> 10 个高频面试问题 + 回答要点 + 完整话术

## Q1：请介绍一下这个项目

### 回答要点

- 一句话定位：人机协同 AI 理财辅助工具
- 四大原则：人机协同、AI 边界、可解释、分层风控
- 技术栈：Vue 3 + FastAPI + LSTM/RF + DeepSeek
- 核心差异化：合规闭环 + 审计溯源 + 隐私优先

### 完整话术

> 「灵析是我独立开发的一款 AI 理财辅助工具，定位是**人机协同**——AI 负责数据处理和信息提炼，投资决策由用户自己做。
>
> 技术上采用四层架构：Vue 3 前端、FastAPI 服务层、LSTM + 随机森林模型层、Tushare + SQLite 数据层。核心功能包括持仓诊断、周期分析、情绪纠偏、场景化投教、分析溯源和合规风控。
>
> 相比市面上的 AI 投顾 Demo，灵析的三个差异化是：**第一**，规则引擎前置拦截，合规闭环有日志有统计；**第二**，每次诊断 7 步审计全链路可追溯；**第三**，持仓 AES-256 本地加密，隐私优先不上云。」

## Q2：为什么选择 LSTM + 随机森林，而不是直接用 LLM？

### 回答要点

- 任务性质不同：时序预测 vs 分类 vs 文本生成
- LSTM 适合价格序列的趋势捕捉
- RF 适合多特征风险分类，可解释性强
- LLM 仅用于投教问答和诊断追问（文本场景）
- 规则引擎兜底，避免模型单点故障

### 完整话术

> 「这是基于**任务匹配**的设计决策。
>
> 持仓诊断涉及三类任务：一是价格趋势判断，这是典型的时序问题，LSTM 用 30 日窗口预测 5 日趋势，比 LLM 更适合；二是风险分级，随机森林用 volatility、PE 分位等 4 维特征做低/中/高分类，可解释性比黑盒 LLM 好；三是文本问答，这才交给 DeepSeek。
>
> 三层互补：LSTM 给趋势参考，RF 给风险标签，规则引擎兜底——比如 RF 超时或模型文件缺失时，自动回退到『亏损超过 5% 判高风险』的规则逻辑，保证系统永远有输出。
>
> 如果面试官追问为什么不用 Transformer：在我们的数据规模下（HS300 成分股、2 年日线），LSTM 训练 30 epoch 即可收敛，RMSE 356、MAE 129，性价比更高。」

## Q3：合规规则引擎是怎么设计的？

### 回答要点

- 11 个禁止词库
- 前置拦截（LLM 调用之前）
- 双写日志（compliance_logs + audit_logs）
- 统计 API 聚合展示
- 三层风控：数据层 / AI 输出层 / 交互层

### 完整话术

> 「合规是灵析的核心设计约束，不是后期补丁。
>
> 规则引擎在 `rule_engine.py` 中维护 11 个禁止词，包括『荐股』『能买吗』『保证收益』等。关键是**前置拦截**——在 `diagnose_chat.py` 中，用户追问先过 `check_question()`，命中禁止词直接返回 403，**零 LLM 调用**。
>
> 拦截事件双写日志：`compliance_logs` 记录拦截详情，`audit_logs` 记录步骤名为『合规问答拦截』。合规面板通过 `/api/admin/compliance-stats` 动态聚合展示拦截次数和规则通过率。
>
> 此外，Compliance 页面明确展示三层风控：数据层做脱敏和异常值拦截，AI 输出层做禁止词和 PE 数值校验，交互层做风险提示弹窗和免责声明。」

## Q4：审计溯源是怎么实现的？为什么需要 7 步？

### 回答要点

- 7 步对应诊断流水线每个阶段
- SQLite audit_logs 持久化
- trace API 查询 + lineage DAG 可视化
- 完整性校验 is_complete
- 满足「可解释性原则」

### 完整话术

> 「可解释性是我们四大设计原则之一。用户有权知道 AI 结论是怎么来的。
>
> 每次诊断在 `portfolio.py` 中按 7 步写入 `audit_logs`：请求接收 → 数据获取 → 数据清洗 → LSTM 预测 → RF 评估 → 规则校验 → 结果生成。前端溯源页通过 `GET /api/trace/{request_id}` 拉取日志，并用 ECharts 渲染 DAG 血缘图，节点分为数据源、处理、模型、输出四类。
>
> 后端还做了完整性校验：`is_complete = len(logs) >= 7`，少于 7 步会标记为不完整，便于排查流水线中断。
>
> 这 7 步不是随意定的，而是和实际代码执行顺序一一对应——面试官如果想验证，我可以打开代码指给他看每一步的 `record_audit_step` 调用。」

## Q5：持仓数据安全是怎么保障的？

### 回答要点

- AES-256 本地加密
- 不上传云端
- 诊断 API 最小化传输
- 旧数据自动迁移
- 一键清除

### 完整话术

> 「隐私优先是 PRD 阶段就定的原则。
>
> 持仓数据通过 `portfolioCrypto.ts` 用 CryptoJS AES-256 加密，密钥由 SHA256 派生，存储格式为 `enc:v1:{ciphertext}`，写入浏览器 localStorage。用户导入时需勾选数据安全承诺书。
>
> 诊断时，前端只向 API 发送资产代码和用户自填的成本价/现价，**不上传完整持仓 JSON**。API Key（Tushare、DeepSeek）全部在服务端 `.env` 中，前端零接触。
>
> 还做了两个细节：一是旧版明文 JSON 自动迁移为加密格式；二是提供『清除本地数据』按钮，一键删除持仓和诊断缓存。」

## Q6：模型灰度发布是怎么做的？

### 回答要点

- NEW_MODEL_RATIO=5 环境变量
- 稳定哈希分流
- seed 设计（request_id / ts_code）
- 响应带 model_variant 标记
- model_manager 版本归档

### 完整话术

> 「我们实现了最小可用的模型灰度方案。
>
> 配置项 `NEW_MODEL_RATIO=5`，表示约 5% 流量走新模型。分流算法是 `abs(hash('lingxi:' + seed)) % 100 < ratio`，用稳定哈希保证同一 request_id 或 ts_code 每次路由一致。诊断用 request_id 做 seed，周期分析用 ts_code。
>
> 响应中会返回 `model_variant: 'new' | 'legacy'` 和 `model_routing.gray_ratio`，前端和日志都能感知当前走的哪条路径。
>
> 训练完成后，`model_manager.register_model_version()` 自动归档到 `models/versions/{type}/{timestamp}/`，元数据写入 `model_metadata.json`。回滚函数已实现，HTTP 管理 API 是 v1.1 计划。」

## Q7：诊断 API 为什么有时候很快、有时候很慢？

### 回答要点

- 快速路径：用户已填本地价格 → 跳过 Tushare + LSTM
- 慢路径：需拉 Tushare 行情 + 跑 LSTM/RF
- Tushare 限流问题
- 前端 45s 超时 + localStorage 缓存 + fallback mock

### 完整话术

> 「这是一个真实的性能优化案例。
>
> 诊断 API 有两条路径：**快速路径**下，用户已在导入页填写成本价和现价，后端检测到 `has_local_prices=true`，跳过 Tushare 拉取和 LSTM 推理，响应时间约 0.17 秒；**慢路径**下需要调 Tushare daily_basic 等接口，加上 LSTM/RF 推理，可能超过 120 秒，还会触发 Tushare 限流。
>
> 我的优化策略是：第一，后端识别本地价格走快速路径；第二，前端设 45 秒超时，超时后展示 fallback 诊断结果；第三，成功结果缓存到 localStorage，二次进入秒开。
>
> 这体现了**性能与功能的 trade-off**——不是每次都跑全量模型，而是根据数据可用性智能降级。」

## Q8：如果让你重新设计，你会改什么？

### 回答要点

- 展示思考深度，不贬低现有方案
- 提 2~3 个具体改进
- 区分「现在能做」和「需要资源」

### 完整话术

> 「整体架构我不大改，但有三处会优化：
>
> **第一**，反馈闭环。`feedback.py` 已实现但未挂载到 `main.py`，v1.1 会注册路由，把用户『有帮助/需改进』反馈写入 DB，驱动模型和 prompt 迭代。
>
> **第二**，模型服务化。目前 LSTM/RF 是在 API 进程内同步推理，高并发时会阻塞。会用 Celery 或 FastAPI BackgroundTasks 做异步推理，前端轮询结果。
>
> **第三**，PostgreSQL 替代 SQLite。审计日志和合规日志会持续增长，SQLite 单写者限制在并发场景下是瓶颈。Docker Compose 已预留 PostgreSQL 配置注释。
>
> 这三项都是工程化演进，不影响当前核心链路的完整性。」

## Q9：这个项目的技术难点是什么？你怎么解决的？

### 回答要点

- 选 1~2 个真实难点
- 描述问题和解决过程
- 量化结果

### 完整话术

> 「最大的难点是**诊断 API 超时导致前端卡死**。
>
> 现象：用户点击诊断后一直显示『分析中...』，后端日志显示 Tushare 限流 + LSTM 推理超过 120 秒。排查发现：每次诊断都对所有资产调 Tushare daily_basic，积分消耗极快；且 TensorFlow 首次加载模型需要 10+ 秒。
>
> 我的解决步骤：第一，后端加 `has_local_prices` 快速路径，跳过 Tushare 和 LSTM，响应降到 0.17s；第二，RF 超时回退规则引擎，避免无限等待；第三，前端 45s 超时 + localStorage 缓存 + mount 时立即展示 fallback。
>
> 结果：主路径从『永远 loading』变为『秒级响应』，用户体验质变。这个过程让我理解了**全链路性能优化**——不是只优化模型，而是从前端超时、后端降级到数据源限流都要考虑。」

## Q10：这个项目的商业价值是什么？

### 回答要点

- 用户价值：省时间、减损失、学知识
- 产品价值：合规可审计、可部署
- 量化估算（诚实标注为演示模型）
- B 端场景：金融机构 AI 辅助工具参考架构

### 完整话术

> 「灵析的商业价值分三层：
>
> **用户层**：首页商业价值计算器估算，单用户年节省约 156 小时分析时间、减少 8~12 次冲动交易、避免 5,000~10,000 元情绪化损失。这些是参数化演示模型，面试中我会诚实说明。

>
> **B 端层**：四层架构 + 三层风控 + 7 步审计的设计模式，可以直接复用到金融机构的 AI 辅助工具合规审查场景——我们卖的不是『预测能力』，而是『合规可审计的 AI 辅助框架』。
>
> 如果做 MVP 验证，我会先聚焦『持仓诊断 + 合规拦截』两个 Must 功能，用真实用户反馈驱动迭代。」
